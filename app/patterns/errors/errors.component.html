Validation & Errors

What Are Errors?

Errors occur either when the application fails during a valid action taken by the user (system-generated) or when the user takes an action that is not allowed or problematic (user-generated).

How does Rivet approach errors?

We first aim for prevention through..

Design

The interface should be designed so that it helps users avoid taking actions that could lead to an error in the first place.

Examples: Date range pickers should not allow a user to select an end date before a start date, Allow a user to select from a drop down menu vs. manually entering an option when there is a limited set to choose from. Ensure an advisor cannot be deleted until clients are reassigned to prevent clients from being stuck in limbo

Validation

Use validation on inputs to prevent user errors and to ensure accurate, clean data is sent to the application. Whenever possible validation should occur "automatically" before the user clicks the submit button.

Examples: Verify that new template names do not conflict with. Validate custom report content to prevent code injection

But, when errors do happen

Identify

Utilize the proper Rivet components to identify the error as close to its context as possible.

Examples: Display input-specific errors in the same space and with the same styling as validation. Use a snack bar component that takes the user to the page with the error when it is occurring out of same page context

Inform

Provide the user with a simple description of what went wrong and what they need to do to continue. Ideally this will involve resolving the issue. If the issue cannot be immediately resolved, (as in the case of a total system failure) the user should be made aware.

Examples: Inclusion of directions to contact support (when appropriate) "Friendly" explanation of resource not found, and letting the user know that they will be directed back to the results page via an error modal

Preserve

If an error is returned, make every attempt to preserve as much of the user's work as possible.

Examples: Retain all input values as set by user. Allow them to continue working from the same point they were prior to the error (or as close as possible)

Inline Error

How should they work?

They allow the user to:

  • continue viewing the data on the current screen
  • choose to ignore the error if it is not preventing them from completing their task
  • easily identify the locations of the errors on the screen within context

Inline errors may also allow a user to resubmit the previous request to attempt to reload the data submitted or re-request the data requested.

Where are they used?
  • Lists
  • Form inputs
  • Form submit buttons
  • Modals
  • Inline edit fields

Error Modal

How should they work?

An error modal should appear when a requested page or link is not available. It should then link the user back to the most recent "safe" area where the user can continue normal use. They are used when a workflow needs interruption.

Where are they used?

Error modals are used when a user tries to view:

  • A page they do not have access to
  • Content that has been deleted, but the link still appears on screen

When not to use error modals

Error modals should not be used to indicate errors that are appearing on the same page or for errors identified by validation. If the user can continue working while the error is displayed on the page, a modal is not needed.

Error Snack Bar

How should they work?

Error snack bars should

  • Appear at the bottom of the screen when the error occurs
  • Allow the user to click a button to be taken to a location where the error can be corrected
  • Remain displayed until the user dismisses them

Users may continue working while an error snack bar is displayed

Where are they used?

An error snack bar should be used when an error occurs out of context - meaning that the problem is not attached to something that the user sees on the screen. The error is not interrupting the user's workflow, but may mean that a previously attempted action was not successfully completed.

When not to use error snackbar:

Do not use an error snack bar when the error can be displayed in context on the same page.

Full Page Error

How should they work?

A full page error should prevent a user from accessing any part of the application. The user's only option is to allow redirection to a different page. It should indicate that the system is down or that the connection to the system has been lost.

Where are they used?

This is used for full 500 "fatal" errors - full system failure.

When not to use full page errors

Do not use alternate full page errors for situations in which page content is not found, or when access to the page is denied. Instead treat those the same way and redirect the user to the login page (if functioning). In general, when content is not found while a user is logged in, they still should be able to see the basic header, footer, and other page elements. These should not be replaced with a full page error except in the case of a total system failure. Instead, use an alternate error component as outlined above.

What kind of language do we use in error messages?

DO:

  • Use the NaviPlan Sage tone of voice
  • Explain in simple terms what went wrong
  • Let the user know what they can do to resolve the issue
  • Encourage the user to contact support if the issue continues to occur (depending on the error type)

DO NOT:

  • Use language that places blame on the user
  • Include technical terms in the main error message content
